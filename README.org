#+TITLE: Gritcode
#+SUBTITLE: Native coding engine

[[https://gritcode.ai][gritcode.ai]]

GPU-rendered AI coding harness built in C++ with OpenGL, HarfBuzz, and FreeType. No GLFW — windowing, input, and clipboard go straight to Wayland, X11, and Cocoa.

⚠️ *Warning:* Gritcode does not have a permission system. AI agents can execute any command on your machine. Press *Escape* to cancel a running request at any time.

** Features

- *GPU-rendered UI* -- OpenGL 3.3 batched quads, SDF rounded rects, 0% idle CPU
- *HarfBuzz text shaping* -- proper Arabic RTL, ligatures, contextual forms, diacritics
- *Emoji support* -- FontConfig fallback, ZWJ sequences, bitmap scaling
- *Native windowing* -- direct Wayland (xdg-shell + xdg-decoration), X11 (EGL + XIM), and Cocoa (NSWindow + NSOpenGLView). No GLFW.
- *Client-side decorations* -- rounded corners and drop shadow on GNOME and other CSD-only compositors, drawn by an SDF shader compose pass
- *Edge-drag resize, touch input, compositor-paced rendering* on Wayland; XIM UTF-8 text input and clipboard round-trip on X11
- *Multi-provider* -- [[https://opencode.ai/zen][OpenCode]] Zen (pay-as-you-go), OpenCode Go (quality coding models, low monthly price), and Claude ACP (requires [[https://docs.anthropic.com/en/docs/claude-code/overview][Claude Code CLI]])
- *Tool use* -- bash, read_file, list_directory with background execution
- *Secure API key storage* -- system keychain (libsecret on Linux, Keychain on macOS)
- *Session persistence* -- per working directory, automatic restore
- *Real context compaction* -- summary LLM call replaces old history in place; no rolling re-compaction
- *Markdown rendering* -- bold, italic, code blocks, headings, blockquotes, lists

** Is this production ready?

No. This is not even pre-alpha. I use Gritcode every day as my primary coding environment, but every few hours I hit a bug — sometimes small, sometimes not. Things break, things get fixed, things change fast. This is a live project under active development.

If you want to see it stabilize faster, [[https://github.com/sponsors/lszl84][sponsor the project]] 💜

** Quick Start

*** Linux (Ubuntu/Debian)

#+begin_src bash
sudo apt install libfreetype-dev libharfbuzz-dev libfontconfig-dev \
  libcurl4-openssl-dev libsecret-1-dev cmake ninja-build \
  libopengl-dev libegl-dev \
  libwayland-dev wayland-protocols libxkbcommon-dev \
  libx11-dev libxext-dev

git clone https://github.com/lszl84/gritcode.git
cd gritcode
cmake -B build -G Ninja -DCMAKE_BUILD_TYPE=Release -DCMAKE_INSTALL_PREFIX=~/.local
cmake --build build
cmake --install build

grit
#+end_src

At least one of Wayland or X11 must be present; CMake probes both and builds
whichever backend it finds. No =libglfw3= dependency — =window_linux.cpp= talks
to =xdg-shell= / =xdg-decoration= and the X server directly.

*** FreeBSD / GhostBSD

#+begin_src bash
# GhostBSD: install system dev packages first
sudo pkg install -g 'GhostBSD*-dev-*'

# FreeBSD/GhostBSD: install build dependencies
sudo pkg install cmake git pkgconf freetype2 harfbuzz fontconfig \
  curl libsecret cmark wayland wayland-protocols libxkbcommon \
  libX11 libXext

git clone https://github.com/lszl84/gritcode.git
cd gritcode
cmake -B build -DCMAKE_BUILD_TYPE=Release -DCMAKE_INSTALL_PREFIX=~/.local
cmake --build build
cmake --install build

grit
#+end_src

On GhostBSD, the ~GhostBSD*-dev-*~ packages provide the compiler, system
headers, and linker. Without them, nothing compiles — they are not installed
by default. On plain FreeBSD, the dev packages are included with the base
system.

*** macOS

#+begin_src bash
brew install cmake ninja

git clone https://github.com/lszl84/gritcode.git
cd gritcode
cmake -B build -G Ninja -DCMAKE_BUILD_TYPE=Release
cmake --build build

open build/grit.app
#+end_src

On macOS, FreeType, HarfBuzz, and curl are fetched and built from source automatically by CMake. The window backend (=window_macos.mm=) is a thin Cocoa wrapper around =NSWindow= + =NSOpenGLView= — no GLFW dependency.

** Architecture

| File                    | Purpose                                                   |
|-------------------------+-----------------------------------------------------------|
| =app.cpp=               | Main app, providers, streaming, tool use, compaction      |
| =gl_renderer.cpp=       | OpenGL batched renderer, SDF shapes                       |
| =font.cpp=              | HarfBuzz shaping, FreeType raster, fallback               |
| =scroll_view.cpp=       | Block layout, word wrap, selection, caret                 |
| =markdown_renderer.cpp= | cmark AST to styled blocks                                |
| =widgets.cpp=           | TextInput, ComboBox, Button (custom)                      |
| =window_linux.cpp=      | Wayland (xdg-shell + xdg-decoration) and X11/EGL backend  |
| =window_macos.mm=       | Cocoa NSWindow + NSOpenGLView backend                     |
| =csd.cpp=               | Client-side decorations: rounded corners, shadow, titlebar |
| =curl_http.cpp=         | Async HTTP with SSE streaming                             |
| =mcp_server.cpp=        | MCP socket server for agent-driven automation             |
| =session.cpp=           | Per-CWD session persistence                               |
| =keychain.cpp=          | libsecret / Keychain API key storage                      |

** Requirements

- Linux (Wayland or X11), macOS, or FreeBSD/GhostBSD
- CMake 3.24+, Ninja (or Make), C++23 compiler
- Linux: freetype2, harfbuzz, fontconfig, libcurl, libsecret-1, OpenGL + EGL (libopengl-dev + libegl-dev on Debian, mesa on Arch, mesa-libGL-devel + mesa-libEGL-devel on Fedora), at least one of: Wayland stack (wayland-client, wayland-egl, wayland-cursor, wayland-protocols, libxkbcommon) or X11 stack (libX11, libXext)
- FreeBSD/GhostBSD: freetype2, harfbuzz, fontconfig, curl, libsecret, cmark, wayland + protocols + libxkbcommon and/or libX11 + libXext (via =pkg install=)
- macOS: Xcode command line tools (dependencies fetched automatically)

** Contributing

✨ *This is all vibe coded.* ✨

Not a single line of code in this project was typed by hand. Every feature, every fix, every refactor — all driven through AI agents. That's the whole point.

I don't intend to write code manually, and as the project grows the focus will be on building features for more surgical agentic interventions — interaction graphs for code components, visual code browsing, targeted refactoring, debugging, performance profiling. The IDE itself is built the way it's meant to be used.

*Pull requests are not accepted for now.* If you find a bug or have a suggestion, please open an issue — and if you can, run your own AI agent to analyze the problem and include its suggestions in the report. That kind of contribution is worth more than a patch. Thanks! 🙏
